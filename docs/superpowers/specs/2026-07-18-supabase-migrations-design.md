# Supabase CLI Migrations — Design Spec

**Date:** 2026-07-18  
**Status:** Approved

---

## Overview

Replace the manual `scripts/*.sql` workflow with Supabase CLI-managed migrations. All database changes are tracked as versioned migration files, applied automatically in CI on push to `main`, and reproducible locally via `supabase db reset`. The GitHub Actions pipeline replaces Vercel for frontend deployment.

---

## Architecture

- **Migration tool:** Supabase CLI (`supabase db push`)
- **Migration tracking:** Supabase's built-in `supabase_migrations.schema_migrations` table
- **Local dev:** `supabase start` (Docker) — full local Supabase instance
- **CI/CD:** GitHub Actions — migrate then deploy to GitHub Pages on push to `main`
- **Base schema:** Pulled from the remote project via `supabase db pull` (covers tables created directly in Supabase before this migration system was introduced)

## Migration Strategy

### Base Schema

The original tables (`members`, `damage_logs`, `train_schedule`, `ooto`, `kill_list`, `friends_list`) have no `CREATE TABLE` scripts — they were created directly in Supabase. These are captured by running `supabase db pull` against the live project, which generates `20260101000000_base_schema.sql` automatically.

### Migration Fixes

Two scripts require corrections when converting:

- **`vs-points-migration.sql`:** Contains a typo (`FOR SELECTcan t`) — fixed in `20260101000005_vs_points.sql`
- **`train-schedule-settings-migration.sql`:** Contains an `INSERT` row for the default `week_mode` value — the `INSERT` moves to `supabase/seed.sql`; the migration keeps only the `CREATE TABLE`

### pg-cron

`scripts/pg-cron-sync.sql` is **not converted to a migration**. It requires hardcoded `<PROJECT_REF>` and `<SERVICE_ROLE_KEY>` values specific to the Supabase project. It remains a manual setup step documented in the project README, to be run once in the Supabase SQL Editor.

### Seed Data

`supabase/seed.sql` combines:
- The member `INSERT` statements from `scripts/seed-members.sql` (with `ON CONFLICT DO NOTHING` guard)
- The `train_schedule_settings` default row (`INSERT INTO train_schedule_settings (key, mode) VALUES ('week_mode', 'push') ON CONFLICT DO NOTHING`)

`seed.sql` is applied by `supabase db reset` in local dev only. It is not run in CI against production.

## Environment Variables

### Local Dev (`.env.local`)

```env
VITE_SUPABASE_URL=http://localhost:54321
VITE_SUPABASE_ANON_KEY=<local anon key from supabase status>
```

`.env.local` is gitignored. `.env.local.example` documents the required keys.

### Production (GitHub Actions environment)

Set as plain environment variables in the GitHub Actions workflow (not secrets — the anon key is safe to expose):

```
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<production anon key>
```

### CI Secrets (GitHub repository secrets)

| Secret | Where to find it |
|---|---|
| `SUPABASE_ACCESS_TOKEN` | app.supabase.com → Account → Access Tokens |
| `SUPABASE_PROJECT_REF` | Project Settings → General → Reference ID |

## Local Dev Workflow

### One-Time Setup

1. Install [Supabase CLI](https://supabase.com/docs/guides/cli) and Docker Desktop
2. `supabase init` — creates `supabase/config.toml`
3. `supabase link --project-ref <ref>` — links CLI to the remote project
4. Copy `.env.local.example` → `.env.local`
5. `supabase start` — starts local instance; run `supabase status` to get the local anon key and update `.env.local`

### Daily Commands

| Command | Effect |
|---|---|
| `supabase start` | Start local Supabase (Postgres, Auth, Studio) on localhost:54321 |
| `supabase stop` | Stop local instance |
| `supabase db reset` | Wipe local DB, re-apply all migrations, apply `seed.sql` |
| `supabase migration new <name>` | Create a new timestamped migration file |
| `supabase db push` | Push pending migrations to production (normally handled by CI) |

## GitHub Actions Pipeline

**File:** `.github/workflows/deploy.yml`  
**Trigger:** push to `main`

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  migrate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: supabase/setup-cli@v1
        with:
          version: latest
      - run: supabase db push --project-ref $SUPABASE_PROJECT_REF
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
          SUPABASE_PROJECT_REF: ${{ secrets.SUPABASE_PROJECT_REF }}

  deploy:
    needs: migrate
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
        env:
          VITE_SUPABASE_URL: ${{ vars.VITE_SUPABASE_URL }}
          VITE_SUPABASE_ANON_KEY: ${{ vars.VITE_SUPABASE_ANON_KEY }}
      - uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
```

`deploy` runs only after `migrate` succeeds (`needs: migrate`). A failed migration blocks the frontend deploy.

## Notes

- **GitHub Pages base path:** If the GitHub repo is not the user/org root repo (e.g., `github.com/user/AllianceTracker` rather than `github.com/user/user.github.io`), the Vite config needs `base: '/AllianceTracker/'` for assets to resolve correctly.
- **Future migrations:** Use `supabase migration new <descriptive-name>` to create new files. Never edit existing committed migrations — add a new one instead.
- **Production migration safety:** `supabase db push` applies only migrations not yet recorded in `supabase_migrations.schema_migrations`. It is idempotent on already-applied migrations.
